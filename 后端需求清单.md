# Echo 后端 API 需求清单

本文档供后端开发者参考。前端已按此契约封装完毕（`src/services/api.ts`），
后端实现完成后，将 `.env.local` 中 `VITE_USE_MOCK` 改为 `false` 即可无缝切换。

---

## 基础信息

- **Base URL**：由前端环境变量 `VITE_API_BASE_URL` 提供，默认 `http://localhost:3001/api`
- **数据格式**：JSON（`Content-Type: application/json`）
- **跨域**：开发环境前端运行在 `http://localhost:5173`，后端需允许该来源的 CORS；
  或使用前端 Vite 代理（`vite.config.ts` 已配置 `/api` 代理到 `http://localhost:3001`）。
- **字符编码**：UTF-8

---

## 1. 发送聊天消息

`POST /chat`

这是核心端点。前端发送用户消息，后端返回 AI 回复文本 + 透明化面板数据。

### 请求体
```json
{
  "message": "我今天又熬夜了，感觉自己很没用",
  "sessionId": "可选，首次不传，后端建立后返回"
}
```

### 响应体
```json
{
  "reply": "谢谢你愿意告诉我这些。熬夜的夜里，人会更容易对自己苛刻……",
  "mirror": {
    "signals": ["自我怀疑", "疲惫"],
    "strategy": "先验证感受，再用开放式提问引导自我觉察",
    "blindspots": ["可能忽略了用户身边已有的支持系统"],
    "limitation": "这是一个语言模型根据模式生成的回应，并非真实情感体验。"
  },
  "sessionId": "session_xxx"
}
```

### 字段说明
| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `reply` | string | 是 | AI 回复完整文本 |
| `mirror.signals` | string[] | 是 | 检测到的情绪关键词（2-4 个） |
| `mirror.strategy` | string | 是 | 使用的语言学方法说明 |
| `mirror.blindspots` | string[] | 是 | 可能忽略的视角（1-3 条） |
| `mirror.limitation` | string | 是 | 固定限制声明（建议固定文案） |
| `sessionId` | string | 是 | 会话 ID，用于多轮上下文 |

### 实现建议
- `reply` 由 LLM 生成，风格应温暖、陪伴、不急于建议。
- `mirror` 是「素养教育」核心：建议对用户消息做情绪分析 → 提取 signals；
  strategy 由 prompt 决定；blindspots 可由第二次 LLM 调用反思生成。
- `limitation` 固定为：`这是一个语言模型根据模式生成的回应，并非真实情感体验。`

## 2. 获取多版本回复（Conversation Lab）

`POST /lab`

对同一用户消息，生成三种不同风格的回应，供用户对比反思。

### 请求体
```json
{
  "message": "我感觉撑不住了",
  "originalReply": "可选，上一轮的常规回复"
}
```

### 响应体
```json
{
  "versions": [
    { "style": "安慰模式", "description": "温柔陪伴", "text": "我知道现在很难……" },
    { "style": "挑战模式", "description": "温和质疑", "text": "你说\"撑不住了\"——这个判断是基于事实……" },
    { "style": "换框模式", "description": "换个角度", "text": "把\"我又没做好\"换成\"我正在学习做好\"……" }
  ]
}
```

### 字段说明
| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `versions` | array | 固定 3 个元素 |
| `versions[].style` | string | 风格名，建议固定为：`安慰模式` / `挑战模式` / `换框模式` |
| `versions[].description` | string | 简短风格说明：`温柔陪伴` / `温和质疑` / `换个角度` |
| `versions[].text` | string | 该风格的回复文本 |

> `style` 与 `description` 的取值前端有对应配色映射，建议保持上述固定值。

## 3. 获取个人洞察（周报）

`GET /insights`

基于用户历史对话数据生成的周报洞察，前端以「信件」形式呈现。

### 响应体
```json
{
  "mainTheme": "学业压力与自我否定",
  "peakHours": "深夜 23:00 — 01:00",
  "dependencySign": "连续 5 天在凌晨 1 点后使用",
  "reflection": "你似乎一直在寻找确定性。也许这一周，可以试着允许一些\"暂时不知道\"。",
  "completedQuests": 3,
  "journalCount": 5,
  "blindspotCount": 4
}
```

### 字段说明
| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `mainTheme` | string | 本周主要情绪主题；数据不足时返回空字符串触发空状态 |
| `peakHours` | string | AI 使用高峰时段（人类可读） |
| `dependencySign` | string | 依赖迹象提示，措辞温和、非诊断 |
| `reflection` | string | 一句反思引导 |
| `completedQuests` | number | 本周完成现实任务数 |
| `journalCount` | number | 日记总数 |
| `blindspotCount` | number | 用户展开 Mirror 面板（发现盲点）的次数 |

### 空状态约定
当 `journalCount === 0 && completedQuests === 0` 时，前端显示空状态插画。
后端可在此情况下返回各字段为空字符串/0。

## 4. 日记管理

日记内容**完全私密**，不用于 AI 分析。

### 4.1 获取列表
`GET /journal`

响应：
```json
[
  { "id": "j_1", "date": "2026-07-15T22:10:00.000Z", "emotion": "迷茫", "preview": "今天和导师聊了研究方向……" }
]
```
> 按日期倒序返回。`preview` 为内容前 60 字。

### 4.2 新建
`POST /journal`

请求体：
```json
{ "content": "日记全文", "emotion": "平静" }
```
`emotion` 枚举：`焦虑` | `低落` | `平静` | `感激` | `迷茫` | `希望`

响应：新建的日记对象
```json
{ "id": "j_2", "date": "2026-07-17T00:44:00.000Z", "emotion": "平静", "preview": "日记前60字", "content": "日记全文" }
```

### 4.3 删除
`DELETE /journal/:id`

响应：`{ "success": true }`

## 5. 完成现实任务

`POST /quest/complete`

用户在「我的角落」点击「今天我做成了一件小事」时调用。

### 请求体
```json
{ "questId": "uuid-xxx" }
```
> 当前 `questId` 由前端生成（UUID），后端可直接记录或自增计数。

### 响应
```json
{ "success": true }
```

## 通用错误处理

- 返回合适的 HTTP 状态码（4xx / 5xx）。
- 前端对 `/chat` 失败会显示「蜡烛熄灭」重试 UI；其余端点失败会静默降级或显示空状态。
- 响应体可包含 `{ "error": "message" }`，但前端目前不强制解析。

## 数据存储建议（后端自行决定）

- 用户标识：可接入登录系统，或用匿名 `sessionId` / `userId`。
- 对话历史：按 sessionId 存储，供 `/insights` 聚合分析。
- 日记：独立表，标记为私密，不进入 AI 训练 / 分析流程。
- quest 计数：按用户 + 周聚合。

## 盲点次数统计（blindspotCount）

前端在用户点击「为什么这样回？」展开 Mirror 面板时，可视为一次「发现盲点」。
后端如需精确统计，可新增端点 `POST /mirror/view` 由前端上报；当前前端未实现上报，
`/insights` 的 `blindspotCount` 暂由后端自行统计或返回固定值。
